## Typography

Source: audited from existing components and globals.css. Exactly 4 sizes and 2 weights are permitted across the entire project.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 14px (`text-sm`) | 400 regular | 1.5 |
| Body | 16px (`text-base`) | 400 regular | 1.5 |
| Heading | 20px (`text-xl`) | 600 semibold | 1.2 |
| Display | 28px (`text-2xl`) | 700 bold | 1.2 |

Notes:
- Settings page h1 uses `text-2xl font-bold` (28px/700) — maps to Display role
- Settings section h2 uses `text-xl font-semibold` (20px/600) — maps to Heading role
- Error message text uses `text-xl font-semibold` (20px/600) — urgency signalled visually within the type scale; ARIA urgency provided by `role="alert"` on ErrorState
- New components in this phase (`AppStateAnnouncer`, `RetakeGuidance`) must use `text-base` (16px/400) for body copy and `text-xl font-semibold` (20px/600) for button labels where emphasis is needed

## Color

Source: audited from existing components, globals.css, and layout.tsx. Project is black/white high-contrast with semantic red for destructive/error and yellow for warning.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` / `bg-white` | Page background, all surfaces (source: body in layout.tsx) |
| Secondary (30%) | `#f3f4f6` / `bg-gray-100`, `#e5e7eb` / `border-gray-200` | Card borders, divider lines, disabled states |
| Accent (10%) | `#000000` / `bg-black` | Primary CTA buttons, VoiceButton (idle/listening), focus rings, bold text |
| Warning | `#92400e` / `text-yellow-800` on `#fefce8` / `bg-yellow-50` | Retake guidance panel only (source: RESEARCH.md RetakeGuidance pattern) |
| Destructive | `#dc2626` / `bg-red-600`, `#b91c1c` / `text-red-700` | Remove buttons (settings), error message text (ErrorState) |

Accent (`bg-black`) reserved for:
1. Primary action buttons: "Scan Menu" (ScanButton), "Proceed with what I have" (RetakeGuidance)
2. VoiceButton in idle and listening states
3. Focus-visible outline (`outline-black`) on all focusable elements
4. Header logotype text
5. Navigation links ("Back to Home" in settings)

Listening state ring: `ring-blue-600` on VoiceButton — sole use of blue; do not add blue elsewhere.

Dark mode: CSS variables in globals.css define `--background: #0a0a0a` and `--foreground: #ededed` for `prefers-color-scheme: dark`. New components must use semantic Tailwind classes (`bg-white`, `text-black`) or CSS variables, not hardcoded hex, to respect dark mode.

## Component Inventory

New components introduced in Phase 6. Existing components require ARIA audits only — no visual changes.

### New: AppStateAnnouncer

Purpose: Announces app-level state transitions to screen readers (extraction complete, extraction started). Fills the gap identified in the ARIA audit.

Visual contract:
- Always in DOM (never conditionally mounted)
- `sr-only` class — no visible rendering
- `role="status"` + `aria-live="polite"` + `aria-atomic="true"`
- Placed in `page.tsx` before conditional state blocks

Content contract:
- `results` state: `"Menu loaded. {restaurantName ?? 'Restaurant menu'} is ready."`
- `processing` state: `"Reading your menu, {fileCount} photo{s}."`
- All other states: empty string `""`

### New: RetakeGuidance

Purpose: Full-screen guidance panel shown when extraction quality is low. Provides spoken and visual guidance plus retake/proceed actions.

Visual contract:
- Wrapper: `flex flex-col gap-4 py-6`
- ARIA live region (always in DOM): `role="alert"` + `aria-atomic="true"` + `sr-only` — assertive for urgency
- Guidance panel: `text-base text-yellow-800 bg-yellow-50 rounded-lg p-4`
- ScanButton (retake): reuse existing component, full-width — primary action
- "Proceed with what I have" button (shown only when `attemptCount >= 2`):
  - `min-h-[48px] px-6 text-xl font-semibold bg-gray-700 text-white rounded-2xl`
  - `focus-visible:outline focus-visible:outline-4 focus-visible:outline-offset-2 focus-visible:outline-black`
  - `aria-label="Proceed with partial menu data"`

Announcement deduplication: When `attemptCount > 1`, prefix announcement with `"Attempt {n}: "` to prevent VoiceOver iOS from suppressing repeated identical strings.

### New: settings/layout.tsx (server component wrapper)

Purpose: Provides page `<title>` for Next.js route announcer on settings navigation.

Visual contract: Passthrough wrapper only (`<>{children}</>`). No visual output.
Metadata: `title: 'Settings — MenuVoice'`

### Existing: page.tsx title metadata

Add/update to: `title: 'MenuVoice — Scan a Menu'`

## ARIA Interaction Contracts

This phase is primarily an ARIA and interaction hardening phase. The following contracts govern all screen reader behavior.

### Live Region Rules

| Region | Element | Role | aria-live | aria-atomic | Always in DOM | Urgency |
|--------|---------|------|-----------|-------------|---------------|---------|
| ProcessingState sr-only | `div` | status | polite | true | Yes | Informational |
| VoiceStateIndicator sr-only | `div` | status | polite | true | Yes | Informational |
| AppStateAnnouncer (new) | `div` | status | polite | true | Yes | Informational |
| RetakeGuidance sr-only (new) | `div` | alert | (implied assertive) | true | Yes (when retake state active) | Urgent — action required |
| ErrorState | `div` | alert | (implied assertive) | — | No — mounted conditionally | Urgent |

Rule: Never combine `role="alert"` with explicit `aria-live="assertive"` — causes VoiceOver iOS double-speak. Use `role="alert"` alone.

Rule: All live regions that need to fire reliably on iOS VoiceOver must be in the DOM before their content is populated. The RetakeGuidance component renders its sr-only `role="alert"` div unconditionally while the retake state is active.

### Focus Management

| Trigger | Focus Destination | Method |
|---------|------------------|--------|
| App load (idle state) | ScanButton | Natural DOM order — skip link present in layout.tsx |
| Retake state entered | ScanButton inside RetakeGuidance | `autoFocus` on ScanButton's visible button element, or `useEffect` + `ref.focus()` |
| Results state entered | VoiceButton | Natural DOM order after MenuSummary |
| Settings page load | First form input (Allergies) | Natural DOM order |
| Settings → Back to Home | ScanButton | Next.js route navigation, focus to `#main-content` |

### Touch Target Minimums

All interactive elements: minimum 48×48px (WCAG 2.5.5 AAA, A11Y-03).

| Element | Actual Size | Notes |
|---------|------------|-------|
| VoiceButton | 80×80px (`w-20 h-20`) | Exceeds minimum — primary voice action |
| ScanButton | Full width × 56px (`min-h-[56px]`) | Primary capture CTA |
| "Proceed with what I have" | Full width × 48px (`min-h-[48px]`) | Secondary action in retake flow |
| Remove buttons (settings) | 48px height (`min-h-[48px]`) | Destructive inline action |
| Settings icon (header) | 40px (`p-2` + 24px SVG + padding) | Borderline — ensure tap target CSS adds invisible padding |
| "Back to Home" link | Minimum 48px via `py-2 px-4` + font size | Acceptable |

### Keyboard Navigation

Tab order (main page, idle state):
1. Skip to content link (visible on focus)
2. Settings icon (Header)
3. ScanButton

Tab order (main page, results state):
1. Skip to content link
2. Settings icon
3. VoiceButton
4. TextInputFallback (if shown)
5. ScanButton (bottom of page)

All interactive elements must be reachable with Tab alone. No keyboard traps.

## State Machine Contract (AppState)

Phase 6 extends `AppState` with a `retake` variant. This lives in the page-level app state machine, not the voice loop state machine.

| State | Trigger | Visual | ARIA Announcement |
|-------|---------|--------|-------------------|
| `idle` | Initial load, RETRY, RETRY_CAPTURE | ScanButton + RecentSessions | sr-only h1 (VoiceOver focus) |
| `processing` | EXTRACTION_STARTED | ProcessingState spinner | ProcessingState polite region |
| `results` | EXTRACTION_SUCCESS (high quality) | MenuSummary + VoiceButton | AppStateAnnouncer "Menu loaded" |
| `retake` | EXTRACTION_LOW_QUALITY | RetakeGuidance panel | RetakeGuidance alert region (assertive) |
| `error` | EXTRACTION_ERROR | ErrorState | ErrorState role=alert (assertive) |

Transitions from `retake`:
- User taps ScanButton → `RETRY_CAPTURE` → `processing`
- User taps "Proceed with what I have" (attemptCount >= 2) → `PROCEED_ANYWAY` → `results`

New actions: `EXTRACTION_LOW_QUALITY`, `PROCEED_ANYWAY`, `RETRY_CAPTURE`

Low-quality detection threshold: `extractionConfidence < 0.3` OR `warnings.length > 0`

## iOS Safari Interaction Contracts

These constraints govern implementation choices — not purely visual, but they affect interaction design.

| Scenario | Constraint | Implementation |
|----------|-----------|----------------|
| Welcome TTS | Cannot play `<audio>` before user gesture | Chain welcome TTS to first mic button tap, not to `useEffect` on mount |
| Retake guidance TTS | File input onChange is a user gesture chain | Attempt TTS after extraction; use `role="alert"` as reliable fallback if TTS is blocked |
| VoiceButton TTS | Already chained to startListening (user tap) | No change needed |
| Polite announcements during TTS | VoiceOver iOS queues polite indefinitely when audio is playing | Informational state updates (AppStateAnnouncer) use polite — acceptable; they fire when app is quiet |
| Mic permission | Safari re-prompts on every reload | MicPermissionPrompt already handles this — no change needed |
| Camera capture | Must be triggered by direct user tap | ScanButton.tsx pattern is correct — no change needed |
