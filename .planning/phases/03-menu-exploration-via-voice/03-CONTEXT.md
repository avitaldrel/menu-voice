<decisions>
## Implementation Decisions

### Conversation API & System Prompt
- Full menu JSON injected into system prompt (~3-5K tokens) — locked by CLAUDE.md, no summarization
- Streaming API response with sentence buffering — latency critical for voice; Phase 2 TTSClient already supports this
- New `/api/chat` route — separate concern from `/api/menu/extract` and `/api/tts`
- System prompt persona: helpful restaurant guide — warm, concise, menu-expert; keeps answers under 3 sentences unless user asks for more

### Proactive Menu Overview (MENU-05)
- Triggered automatically when app enters `results` state after menu extraction — same moment voice loop auto-starts
- Contains restaurant type + category names + total item count: "This looks like an Italian restaurant. They have Appetizers, Pasta, Entrees, Desserts, and Drinks — 38 items total. What would you like to know about?"
- Generated by Claude as first assistant message in conversation history — Claude generates it from menu JSON, then TTS speaks it
- User can interrupt overview mid-sentence (already supported by Phase 2 interrupt behavior)

### Response Style & Conversation Flow
- Concise item listings: name + one-line description. Lists max 5 items then asks "Want to hear more?"
- No-match queries: acknowledge + suggest closest match from menu
- Prices mentioned only when user asks — don't include price by default in item descriptions
- Full conversation history kept in messages array — context window allows 10-20 turn restaurant sessions without truncation

### Claude's Discretion
- System prompt engineering details (exact wording, temperature, max tokens)
- Streaming chunk handling and sentence boundary detection integration with existing TTSClient
- Conversation state management architecture (where to store messages array)
- Error handling for API failures mid-conversation

</decisions>

- `src/lib/app-state.ts` — AppState with 'results' state containing Menu data

### Established Patterns
- API routes use Anthropic SDK (`@anthropic-ai/sdk`) for Claude calls
- State machines via discriminated unions + useReducer
- ARIA live regions always mounted in DOM for screen reader reliability
- Hooks return action methods and state (useMenuExtraction, useVoiceLoop pattern)

### Integration Points
- `useVoiceLoop` currently echoes transcript back — needs to send to `/api/chat` instead
- Menu data from `appState` (results state) needs to flow into chat system prompt
- Chat API route needs Anthropic SDK streaming (`stream: true`)
- Voice loop `handleTextInput` (Firefox fallback) should use same chat path
