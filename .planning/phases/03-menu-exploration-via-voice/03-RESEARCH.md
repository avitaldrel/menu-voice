## Summary

Phase 3 wires the extracted `Menu` object (already in `appState.results`) into a Claude conversation API, replaces the Phase 2 echo placeholder in `useVoiceLoop`, and generates a proactive menu overview. The core deliverable is `/api/chat` â€” a Next.js App Router streaming route that accepts a messages array plus the serialized menu JSON, calls `client.messages.stream()` from the Anthropic SDK, and pipes `content_block_delta` text chunks back as a plain-text `ReadableStream`. The client side reads these chunks with `fetch` + `response.body.getReader()`, feeds each text delta into the existing `TTSClient.queueText()`, and calls `TTSClient.flush()` at stream end.

The architecture is fully validated by the existing codebase. `TTSClient` already handles sentence buffering and streaming input via `queueText()` / `flush()`. `useVoiceLoop` already has the `triggerResponse` placeholder that must be replaced. The `VoiceState` machine does not need modification â€” `processing` state already covers the in-flight API call, and `FIRST_AUDIO_READY` fires on the first `onSpeakingStart` callback from TTSClient just as before.

The main new surface area is: (1) the `/api/chat` route with streaming response, (2) conversation history management inside `useVoiceLoop` (a `useRef` shadow array so closures never see stale history), and (3) the proactive overview trigger (a new `useEffect` in `page.tsx` that fires the first chat call without user speech when `appState` transitions to `results`).

**Primary recommendation:** Build `/api/chat` using `client.messages.stream()` + `ReadableStream` piping; consume client-side with `fetch` + `getReader()`; store conversation history in a `useRef` array inside `useVoiceLoop` and mirror to `useState` for rendering.

### Pattern 1: Streaming Route Handler (`/api/chat`)

**What:** POST route that takes `{ messages, menu }`, calls `client.messages.stream()`, pipes text deltas as plain-text `ReadableStream`.

**When to use:** Every user voice query (and the initial proactive overview).

**Example:**
```typescript
// src/app/api/chat/route.ts
// Source: Anthropic SDK MessageStream.ts + Next.js App Router streaming docs
import Anthropic from '@anthropic-ai/sdk';
import type { Menu } from '@/lib/menu-schema';

export const maxDuration = 30;
export const dynamic = 'force-dynamic';

interface ChatMessage {
  role: 'user' | 'assistant';
  content: string;
}

export async function POST(request: Request) {
  if (!process.env.ANTHROPIC_API_KEY) {
    return Response.json({ error: 'ANTHROPIC_API_KEY not configured' }, { status: 500 });
  }

  let body: { messages: ChatMessage[]; menu: Menu };
  try {
    body = await request.json();
  } catch {
    return Response.json({ error: 'Invalid request body' }, { status: 400 });
  }

  const { messages, menu } = body;
  const client = new Anthropic();

  const systemPrompt = buildSystemPrompt(menu);

  const stream = client.messages.stream({
    model: 'claude-sonnet-4-6',
    max_tokens: 512,
    system: systemPrompt,
    messages,
  });

  const readable = new ReadableStream({
    async start(controller) {
      const encoder = new TextEncoder();
      stream.on('text', (textDelta) => {
        controller.enqueue(encoder.encode(textDelta));
      });
      await stream.finalMessage();
      controller.close();
    },
    cancel() {
      stream.abort();
    },
  });

  return new Response(readable, {
    headers: { 'Content-Type': 'text/plain; charset=utf-8' },
  });
}
```

### Pattern 2: Client-Side Stream Consumption (`useVoiceLoop` replacement for `triggerResponse`)

**What:** `fetch` the `/api/chat` route, read text chunks with `getReader()`, pipe each delta into `TTSClient.queueText()`, call `flush()` on stream end.

**When to use:** On every user transcript and for the proactive overview.

```typescript
// Inside useVoiceLoop or a useChatStream helper
// Source: MDN ReadableStream.getReader() + TTSClient API (src/lib/tts-client.ts)

const abortControllerRef = useRef<AbortController | null>(null);

const triggerResponse = useCallback(async (userText: string | null) => {
  // Cancel any in-flight request
  abortControllerRef.current?.abort();
  const controller = new AbortController();
  abortControllerRef.current = controller;

  // Build messages array: append user message if not overview call
  const nextMessages: ChatMessage[] = userText
    ? [...messagesRef.current, { role: 'user', content: userText }]
    : messagesRef.current; // overview: messages is already primed

  try {
    const res = await fetch('/api/chat', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ messages: nextMessages, menu: menuRef.current }),
      signal: controller.signal,
    });

    if (!res.ok || !res.body) {
      dispatch({ type: 'ERROR', message: 'Failed to get response' });
      return;
    }

    const reader = res.body.getReader();
    const decoder = new TextDecoder();
    let fullResponse = '';

    while (true) {
      const { done, value } = await reader.read();
      if (done) break;
      const chunk = decoder.decode(value, { stream: true });
      fullResponse += chunk;
      ttsClientRef.current?.queueText(chunk);
    }
    ttsClientRef.current?.flush();

    // Persist assistant turn into history
    messagesRef.current = [
      ...nextMessages,
      { role: 'assistant', content: fullResponse },
    ];
    setMessages([...messagesRef.current]);
  } catch (err) {
    if ((err as Error).name === 'AbortError') return; // intentional cancel
    dispatch({ type: 'ERROR', message: 'Conversation error' });
  }
}, []);
```

### Anti-Patterns to Avoid

- **Re-creating `AbortController` inside every render:** Creates a new controller on every render; the ref should be mutated, not replaced each render.
- **Storing messages in `useState` only:** Stale closure in the `fetch` callback reads the messages array captured at hook creation time, missing all subsequent turns. Use `useRef` as the authoritative source, `useState` for rendering.
- **Firing both the proactive overview AND `startListening()` simultaneously:** The voice loop auto-start should be gated on TTS playback completing (PLAYBACK_ENDED), not on `results` entry. The existing D-01 useEffect already handles this â€” Phase 3 must integrate the overview trigger without breaking the existing guard logic.
- **Using `stream: true` in `create()` instead of `messages.stream()`:** The high-level `stream()` method handles SSE parsing, connection lifecycle, and abort propagation automatically. The `stream: true` flag on `create()` returns a raw `Stream<RawMessageStreamEvent>` requiring manual event type discrimination.
- **Not calling `stream.abort()` in `ReadableStream.cancel()`:** When the client disconnects mid-response, the server-side stream must be aborted to avoid hanging Anthropic API connections.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SSE event parsing | Custom SSE parser | `client.messages.stream()` with `.on('text', ...)` | SDK handles all event types, errors, abort, backpressure |
| Sentence boundary detection | Custom regex chunker | Existing `TTSClient.queueText()` + `flush()` | Already implemented and tested in Phase 2 |
| Conversation history serialization | Custom format | Standard `{ role, content }` array | Matches Anthropic Messages API exactly; no transformation needed |
| Stream cancellation | Custom timeout/flag | `AbortController` passed to `fetch` + `stream.abort()` on server | Platform-native, cleans up both client fetch and server Anthropic connection |
| Text buffering before TTS | Custom accumulator | `TTSClient.queueText()` handles incomplete sentence fragments | Already handles the partial-sentence buffer between chunks |

**Key insight:** The Phase 2 TTSClient was designed specifically to receive streaming text chunks â€” it already buffers partial sentences and plays complete sentences as soon as they form. The `/api/chat` route simply needs to forward Anthropic text deltas as-is to the client, and the client feeds them directly into the TTSClient pipeline.

## Common Pitfalls

### Pitfall 1: Stale Conversation History in Async Callback

**What goes wrong:** The `fetch` callback inside `triggerResponse` reads `messages` state captured at hook initialization. After the second turn, it sends only the first exchange, losing all prior context.

**Why it happens:** JavaScript closures capture variables by reference at creation time. `useState` setter is stable but the *value* is not â€” callbacks close over the initial value unless re-created via dependency arrays. Re-creating the callback on every messages change defeats the purpose of `useCallback`.

**How to avoid:** Store the authoritative messages array in a `useRef`. The ref's `.current` is always the live value â€” closures that read `messagesRef.current` never see stale data. Mirror to `useState` for rendering by calling `setMessages([...messagesRef.current])` after each update.

**Warning signs:** Conversation loses context after 2 turns; Claude responds as if previous turns never happened.

### Pitfall 2: Race Condition on Rapid Voice Input

**What goes wrong:** User speaks again before the previous response finishes streaming. Two concurrent `fetch` requests race; the earlier response's text chunks arrive after the newer response has started, corrupting the audio queue.

**Why it happens:** `triggerResponse` is called by both the `SpeechManager.onTranscript` callback and potentially by interrupt/restart events. Without cancellation, both requests feed into `TTSClient.queueText()` concurrently.

**How to avoid:** Store `AbortController` in a `useRef`. At the start of every `triggerResponse` call, call `abortControllerRef.current?.abort()` then create a new controller. Catch `AbortError` separately to avoid false error state transitions.

**Warning signs:** TTS speaks garbled/merged responses; conversation history grows with duplicate entries.

### Pitfall 3: Proactive Overview Triggering After User Already Speaks

**What goes wrong:** The `results` state useEffect fires the proactive overview, but the user (or the D-01 auto-start) has already initiated a voice exchange. The overview request conflicts with the first user message.

**Why it happens:** `page.tsx` already has a `useEffect` that auto-starts `startListening()` on `results` entry (D-01). If the overview `useEffect` also fires and both have the same dependency, the ordering is non-deterministic.

**How to avoid:** Sequence the overview and startListening explicitly. The overview call should be triggered from inside `useVoiceLoop` (or a dedicated `triggerOverview` callback) and the voice loop should only begin listening *after* `PLAYBACK_ENDED` fires at the end of the overview TTS â€” using the existing state machine transition (`speaking -> listening`). The D-01 `startListening` useEffect in page.tsx should either be removed or gated on "no overview pending."

**Warning signs:** Voice loop starts listening mid-overview speech; first user transcript is sent concurrently with the overview request.

### Pitfall 4: Server-Side Anthropic Stream Left Open After Client Disconnect

**What goes wrong:** Client closes the browser tab or navigates away mid-stream. The `ReadableStream.start()` async loop continues, and the Anthropic connection stays open until the response finishes.

**Why it happens:** `ReadableStream` `start()` runs independently of client connection state unless a `cancel` method is provided.

**How to avoid:** Implement `cancel()` on the `ReadableStream` that calls `stream.abort()`. This signals the Anthropic SDK's `AbortController`, terminating the upstream connection.

**Warning signs:** Server CPU usage stays elevated after client disconnect; Anthropic usage shows unexpectedly complete responses even when users navigate away.

### Pitfall 5: System Prompt Size vs. Response Budget

**What goes wrong:** Menu JSON is 4K tokens, system prompt scaffolding is ~200 tokens. With `max_tokens: 512` for the response and Claude's model context window, this is fine. But if someone photographs a large tasting-menu-style menu, JSON could reach 8K+ tokens, and `max_tokens: 512` gives Claude too little room to enumerate items.

**Why it happens:** Token budget is fixed at route instantiation. Very large menus are rare but possible.

**How to avoid:** Use `max_tokens: 512` for single-item queries and category listings (per the "under 3 sentences" style rule). Add a note in the system prompt that Claude should be brief. Do NOT increase `max_tokens` dramatically â€” it does not help TTS latency and the streaming model means users hear text as it generates anyway.

**Warning signs:** Claude response is cut off mid-sentence with `stop_reason: "max_tokens"`.

### Pitfall 6: JSON Serialization of Menu Emitting Circular Structures

**What goes wrong:** `JSON.stringify(menu)` throws if the `Menu` object has been mutated by user code to include circular references (unlikely given the schema but worth noting).

**Why it happens:** Standard `JSON.stringify` does not handle circular references.

**How to avoid:** Menu data flows from the extraction response through `appReducer` state â€” it is plain JSON already. No circular reference risk in practice. Use `JSON.stringify(menu, null, 2)` for readable formatting that helps Claude parse structure.

   - What's unclear: Exact sequencing depends on whether the overview is triggered from `page.tsx` or from inside `useVoiceLoop`.
   - Recommendation (Claude's discretion): Trigger the overview from `useVoiceLoop` via an explicit `triggerOverview(menu)` method exposed by the hook. The D-01 `useEffect` in `page.tsx` calls `triggerOverview` instead of `startListening` â€” the voice loop only begins listening after `PLAYBACK_ENDED` fires, which is the natural end-of-overview moment. This removes the need for the existing D-01 `startListening` call entirely (or gates it on overview completion).
