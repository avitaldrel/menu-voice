# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 02: Voice Interface.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none |
| Preset | not applicable |
| Component library | none — hand-rolled Tailwind CSS v4 |
| Icon library | inline SVG (established in Phase 1) |
| Font | Geist Sans (via CSS var `--font-geist-sans`); fallback Arial, Helvetica |

**Rationale:** Phase 1 established a manual Tailwind system. No `components.json` exists. Introducing shadcn mid-project would break visual consistency with existing components (ScanButton, ErrorState, ProcessingState, Header). Phase 2 continues the same hand-rolled approach.

## Spacing Scale

Declared values (multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline label padding |
| sm | 8px | Compact element spacing, icon button padding |
| md | 16px | Default element spacing, section inner padding |
| lg | 24px | Section padding, gap between voice state indicator and controls |
| xl | 32px | Layout gaps between major voice UI zones |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level top/bottom spacing |

Exceptions:
- **Microphone button touch target: 80px minimum** (circle button, accessible for blind users tapping by memory — larger than the 56px Phase 1 minimum)
- **Thinking chime visual indicator: 44px** (icon-size visual, matches WCAG 2.5.5 enhanced target)

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 18px | 600 (semibold) | 1.4 |
| Heading | 20px | 700 (bold) | 1.2 |
| Status/State | 14px | 400 (regular) | 1.4 |

**Notes (source: Phase 1 codebase):**
- Label (18px/600) matches existing button and status text convention (`text-lg font-semibold`)
- Heading (20px/700) matches Header brand name (`text-xl font-bold`)
- Only 2 weights used: 400 and 600/700 — bold only for headings, semibold for all interactive labels
- Status/State (14px/400) used for the ARIA live region visible text and transcript display below microphone button

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` | Page background, voice UI card background |
| Secondary (30%) | `#f9fafb` (gray-50) | Voice state indicator panel, transcript container |
| Accent (10%) | `#000000` (black) | Microphone button (active/listening state), primary action surfaces |
| Listening indicator | `#2563eb` (blue-600) | Microphone button pulse ring during listening state only |
| Speaking indicator | `#16a34a` (green-600) | Audio waveform or speaking dot animation during speaking state only |
| Error/Destructive | `#b91c1c` (red-700) | Error state text and error border (matches Phase 1 `text-red-700`) |

Accent (black) reserved for:
- Microphone button fill in idle and active states
- Primary CTA button (consistent with Phase 1 ScanButton)

Listening indicator (blue-600) reserved for:
- Pulse ring animation around microphone button during `listening` state
- ARIA state label text when state is "Listening..."

Speaking indicator (green-600) reserved for:
- Animated speaking dot or waveform icon during `speaking` state
- ARIA state label text when state is "Speaking..."

Processing/thinking indicator:
- Use existing `border-black border-t-transparent rounded-full animate-spin` (matches Phase 1 ProcessingState spinner) for the visual `processing` state
- Chime audio cue plays alongside — no additional color change needed

Dark mode:
- Background flips to `#0a0a0a`, foreground to `#ededed` (established in globals.css)
- Blue-600 and green-600 remain unchanged (sufficient contrast on dark backgrounds)
- Microphone button: inverted to `bg-white text-black` in dark mode

## Voice State Indicator

This is Phase 2's primary new visual component. The contract for each state:

| State | Label Text | Visual | Color Cue |
|-------|-----------|--------|-----------|
| `idle` | "Tap to start listening" | Static mic icon in black circle | Black |
| `listening` | "Listening..." | Mic icon + pulsing blue ring | Blue-600 ring |
| `processing` | "Thinking..." | Spinning ring (existing Phase 1 pattern) | Black spinner |
| `speaking` | "Speaking..." | Animated speaking dots (3-dot bounce) | Green-600 dots |
| `error` | "Something went wrong" | Static mic icon, muted state | Red-700 label |

The ARIA live region (`role="status" aria-live="polite"`) announces state label text on every state transition. This is an invisible region — the visible label below the button provides the visual equivalent.

## Microphone Button

The microphone button is the primary interaction element for Phase 2:

- **Shape:** Circle, 80px diameter
- **Touch target:** 80px x 80px minimum (no padding reduction)
- **Icon:** Inline SVG microphone, 32px, `aria-hidden="true"`
- **Idle:** `bg-black text-white` circle
- **Listening:** `bg-black text-white` circle + `ring-4 ring-blue-600 ring-offset-2 animate-pulse`
- **Processing:** `bg-gray-400 text-white` circle (disabled appearance) + spinner shown separately
- **Speaking:** `bg-gray-400 text-white` circle (tap to interrupt — destructive — see Copywriting)
- **Error:** `bg-gray-200 text-gray-600` circle
- **Focus:** `focus-visible:outline focus-visible:outline-4 focus-visible:outline-offset-2 focus-visible:outline-black` (matches Phase 1 pattern)
- **aria-label:** Dynamic — changes with state:
  - Idle: "Tap to start listening"
  - Listening: "Listening for your question. Tap to stop."
  - Processing: "Thinking. Please wait."
  - Speaking: "Speaking your answer. Tap to stop."
  - Error: "Error occurred. Tap to try again."

## Text Input Fallback (Firefox / No Web Speech API)

Displayed only when `window.SpeechRecognition` and `window.webkitSpeechRecognition` are both absent.

- **Message:** "Voice recognition is not available in this browser. Try Chrome or Safari for the full experience."
- **Style:** `role="alert"`, `bg-yellow-50 border border-yellow-200 rounded-lg p-4` (matches Phase 1 warning style)
- **Text input:** Standard `<input type="text">` with `min-h-[48px]` and submit button labeled "Send"
- **Font:** 16px/400 (body)

---

## Mic Permission Pre-Prompt

Shown once before the browser's native permission dialog fires, so blind users know what is about to happen:

- **Copy:** "MenuVoice needs your microphone to hear your questions. When prompted, tap Allow."
- **Role:** `role="status"` (not alert — informational, not urgent)
- **Style:** `bg-gray-50 border border-gray-200 rounded-lg p-4 text-base` with a dismiss button
- **Dismiss button:** "Got it" — `min-h-[48px] px-6 text-lg font-semibold bg-black text-white rounded-2xl`

---

## Transcript Display

Shows the user's last spoken utterance while processing, and assistant response text while speaking:

- **Placement:** Below microphone button, inside a `bg-gray-50 rounded-lg` container
- **User utterance:** Prefixed with "You:" in `font-semibold`, 16px/400, `text-gray-900`
- **Response text (while speaking):** 16px/400, `text-gray-700`, streams in as sentences are spoken
- **Max height:** `max-h-48 overflow-y-auto` — scrolls for long responses
- **aria-live:** `aria-live="off"` — the audio TTS carries the content; visual is supplementary only

**Destructive actions in this phase:**
- Tapping the microphone button during `speaking` state interrupts active TTS playback. This is the only destructive action. Approach: immediate action, no confirmation, no undo. The state machine transitions to `idle` and the user can speak again. Button aria-label during speaking state pre-announces this: "Speaking your answer. Tap to stop."

## State Machine Interaction Contract

The voice loop state machine has the following allowed transitions. The executor must enforce these — no simultaneous mic + TTS active:

```
idle          → listening    (user taps mic button or auto-start after menu load)
listening     → processing   (speech recognition fires result event)
listening     → idle         (user taps mic button to stop, or 3s silence timeout)
listening     → error        (speech recognition error event)
processing    → speaking     (first TTS audio segment ready)
processing    → error        (API failure)
speaking      → listening    (audio playback ends — auto-restart)
speaking      → idle         (user taps mic button to interrupt)
error         → idle         (user taps mic button to retry)
```

Auto-restart: after `speaking → listening` transition, the mic restarts automatically. There is no re-tap required.

## Accessibility Contract

All requirements are HARD REQUIREMENTS for this phase (primary user is blind):

1. ARIA live region: `role="status" aria-live="polite" aria-atomic="true"` always in DOM (never conditionally mounted). Content changes on state transition. — source: Phase 1 ProcessingState pattern, CONTEXT.md decision
2. TTS audio element: `<audio>` element only. `window.speechSynthesis` is the fallback only, never primary. — source: CLAUDE.md critical decision, CONTEXT.md
3. Mic button aria-label: Must update dynamically with state (see Microphone Button section above)
4. No simultaneous speech recognition + TTS playback — state machine enforces this — source: VOICE-04, CONTEXT.md
5. Thinking chime: soft repeating tone every 2s during `processing` state — non-intrusive audio feedback — source: CONTEXT.md decision
6. Pre-mic-permission prompt: announced before browser dialog — source: CONTEXT.md decision
7. Touch targets: microphone button minimum 80px circle; all other buttons minimum 48px height
