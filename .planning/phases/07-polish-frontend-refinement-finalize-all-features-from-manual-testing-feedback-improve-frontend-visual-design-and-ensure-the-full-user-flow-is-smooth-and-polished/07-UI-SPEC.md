| Property | Value |

## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, badge padding, inline tight spacing |
| sm | 8px | Button internal padding, compact list gaps |
| md | 16px | Default element spacing, card padding |
| lg | 24px | Section padding, vertical gaps between UI zones |
| xl | 32px | Layout gaps between major sections |
| 2xl | 48px | Hero button vertical padding, major section breaks |
| 3xl | 64px | Page-level top/bottom breathing room |

Exceptions:
- Touch targets: minimum 48px height on all interactive elements (A11Y-03 hard requirement — current `min-h-[72px]` on CTA buttons is correct and must be preserved)
- VoiceButton: 96px diameter (w-24 h-24 = 96px) — established in Phase 2, do not reduce
- Settings gear icon tap target: minimum 44px (upgrade from current 24px icon + 8px padding = 40px; pad to 44px minimum for D-13)

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px | 400 (regular) | 1.5 | TranscriptDisplay text, category item descriptions, guidance copy |
| Label | 14px | 400 (regular) | 1.4 | Metadata (item count, menu type label, status indicator text) |
| Heading | 24px | 600 (semibold) | 1.2 | Restaurant name (MenuSummary h2), category headers, state headings |
| Display | 32px | 600 (semibold) | 1.1 | Welcome "Scan My Menu" button label, hero action labels |

**Constraints:**
- Exactly 2 weights: 400 (regular) and 600 (semibold). Visual differentiation between Heading and Display roles is achieved through size (24px vs 32px), not weight variation.
- Body size must stay at 16px minimum — accessibility requirement for low-vision users (A11Y-03).
- No 12px or 13px text anywhere in the UI — smallest permitted size is 14px (label role).

## Color

### Design Token Definitions (globals.css `@theme` block)

```css
@theme {
  /* Dominant surface — warm off-white cream (D-02) */
  --color-background: oklch(97% 0.015 85);

  /* Primary foreground — warm near-black */
  --color-foreground: oklch(22% 0.01 85);

  /* Secondary surface — warm light muted (cards, panels, nav) */
  --color-muted: oklch(93% 0.01 85);
  --color-muted-foreground: oklch(50% 0.01 85);

  /* Accent — teal/sage for primary interactive elements (D-01) */
  --color-accent: oklch(55% 0.12 185);
  --color-accent-hover: oklch(48% 0.12 185);
  --color-accent-foreground: oklch(99% 0 0);

  /* Destructive — muted warning red for retake guidance panels */
  --color-destructive: oklch(55% 0.18 25);
  --color-destructive-foreground: oklch(99% 0 0);

  /* Warning surface — retake guidance, extraction warnings */
  --color-warning-surface: oklch(95% 0.05 85);
  --color-warning-border: oklch(80% 0.10 85);
  --color-warning-text: oklch(40% 0.08 65);
}
```

**Source:** RESEARCH.md Pattern 1 — OKLCH values directly from research; hue 85 = warm yellow-orange, hue 185 = teal-cyan, hue 25 = warm red.

### 60/30/10 Color Contract

| Role | Token | Value | Usage |
|------|-------|-------|-------|
| Dominant (60%) | `--color-background` | `oklch(97% 0.015 85)` — warm cream | Page background, body, all screen surfaces |
| Secondary (30%) | `--color-muted` | `oklch(93% 0.01 85)` — warm light gray | TranscriptDisplay container, category expand panels, header background, card surfaces |
| Accent (10%) | `--color-accent` | `oklch(55% 0.12 185)` — teal/sage | Reserved-for list below |
| Destructive | `--color-destructive` | `oklch(55% 0.18 25)` — muted red | Retake flow "Scan New Menu" after 2+ failed attempts only |

**Accent reserved for (explicit list):**
1. VoiceButton background in `idle` state — the one dominant interactive element on the results screen
2. ScanButton background — the one dominant interactive element on the idle/welcome screen
3. Welcome "Scan My Menu" button background
4. Header "MenuVoice" logo teal portion (updates D-03 text logo to use accent teal on "Voice")
5. Focus ring color on all focusable elements (replaces current `focus-visible:outline-black`)
6. VoiceButton `listening` ring — replaces current `ring-blue-600` with `ring-accent`

**Accent is NOT used for:** secondary actions, metadata text, expanded category chevrons, disabled states, borders, dividers, or background surfaces.

### Body Background Migration

`layout.tsx` currently has `bg-white` hardcoded on `<body>`. This must be replaced with `bg-background text-foreground` to pick up the `--color-background` CSS variable (RESEARCH.md Pitfall 2).

### Dark Mode

Defer to v2. Phase 7 targets light mode only. The existing `@media (prefers-color-scheme: dark)` block in globals.css is left unchanged but does not receive the new warm palette.

## State Transitions

**Source:** CONTEXT.md D-07, D-09, D-10; RESEARCH.md Pattern 2.

### Fade-In Wrapper (FadePanel component)

All major state panels (welcome, idle, processing, results, retake, error) wrap their content in a `FadePanel` component:

```
mount → opacity-0 → requestAnimationFrame → opacity-100
duration: 300ms
easing: ease-in-out
Tailwind classes: transition-opacity duration-300 ease-in-out
```

- Error and retake states use `duration-200` (faster — avoids artifact on quick retry, RESEARCH.md Pitfall 3)
- Welcome → idle transition uses `duration-400` (success path, more deliberate)
- Results state uses `duration-300`

### Fade-Out Before State Dispatch

For transitions out of a state (before dispatch), delay the reducer action by 150ms to allow opacity-0 to complete:

```
user action → set opacity-0 → setTimeout 150ms → dispatch reducer action → new state mounts with fade-in
```

Only applied to intentional user-initiated transitions (Scan My Menu tap, Scan tap, Retry tap). Not applied to automatic transitions (extract success → results) — those use fade-in on arrival only.

### prefers-reduced-motion

All transitions disabled when `prefers-reduced-motion: reduce`:

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    transition-duration: 0.01ms !important;
    animation-duration: 0.01ms !important;
  }
}
```

Add to `globals.css`. This satisfies WCAG 2.3.3 (W3C C39 technique). ARIA live regions are never animated (RESEARCH.md Pitfall 4).

### TTS Fade-Out for Interrupts (D-10)

When user taps to interrupt speaking:
- `audioElement.volume` ramps from current volume to 0 over 200ms (10 steps × 20ms)
- After fade completes: `audio.pause()`, reset `audio.volume = 1`
- Brief 300ms pause (existing `setTimeout` in `handleScreenTap`)
- Then `startListening()` activates

This happens entirely within `TTSClient.stop()`. Does NOT use SpeechSynthesis (CLAUDE.md hard constraint).

**Source:** CONTEXT.md D-17; RESEARCH.md Pattern 5.

| Spoken phrase | Action | TTS confirmation |
|---------------|--------|-----------------|

## Hint System (D-16, D-18)

**Source:** CONTEXT.md D-16, D-18; RESEARCH.md Pattern 6.

Session count tracked in IndexedDB `settings` store under key `'sessionCount'`. No DB version bump required.

| Session count | Behavior |
|---------------|----------|
| 1 (first visit) | Full spoken tutorial: "Welcome to MenuVoice! You can ask me about any item on the menu, get recommendations, or ask about ingredients. Just tap the microphone and speak naturally." Plays after the overview TTS. |
| 2–5 | Occasional contextual hint: 30% chance per conversation turn, max 1 hint per session. Example: "You can say 'open settings' to manage your food allergies." |
| 6–10 | Rare hint: 10% chance per turn, max 1 per session. |
| 11+ | Very rare: 5% chance per turn, max 1 per session. |

Hints are appended to the overview TTS in `useVoiceLoop` (not added to system prompt). Hint text tone: natural conversation, not tutorial-speak. "By the way, you can..." not "HINT: To navigate to...".

## Accessibility Constraints

These are non-negotiable and carry forward from all prior phases:

1. All interactive elements: minimum 48px touch target height (44px minimum width for icon-only targets)
2. ARIA live regions (`AppStateAnnouncer`, `VoiceStateIndicator`, `ProcessingState` hidden div) are ALWAYS in DOM — never conditionally mounted or animated
3. Processing message ARIA region stays static — only the visible display cycles (prevents screen reader spam every 4 seconds)
4. Focus rings use `focus-visible:` not `focus:` — avoid showing rings on mouse taps
5. TTS output uses `<audio>` element only — never SpeechSynthesis (fixes two existing bugs: welcome TTS in handleStart + processing reminder interval)
6. `prefers-reduced-motion: reduce` disables all transitions/animations
7. Never put `aria-live` AND `role=alert` on the same element — `role=alert` implies assertive; double declaration causes VoiceOver iOS double-speak
